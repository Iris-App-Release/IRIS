## How git fits into the workflow

All future changes should follow this pattern:

```
# After any meaningful change
git add <specific files>
git commit -m "short description of what changed and why"
git push
```

There is no branching strategy yet. A single `main` branch is appropriate until
the project has multiple concurrent workstreams (e.g. a separate signing/release
track alongside active development).

**Before a build:** commit any source changes so the git state matches the DMG
you are about to produce. This makes `git log` a reliable changelog.

**After a build:** tag the commit with the version number:
```bash
git tag v1.5
git push origin v1.5
```

When GitHub Releases is set up (see [[productification]] Milestone 2), the DMG
will be attached to the matching tag as a release asset.

## GitHub Releases (planned)

Once DMGs are notarized (Milestone 1 in [[productification]]), the distribution
workflow will be:

1. `bash Build/build_dmg.sh` → produces `Build/Iris-<version>.dmg`
2. `git tag v<version> && git push origin v<version>`
3. On GitHub: create a Release from that tag, attach the DMG
4. Users download from the Releases page — a stable, versioned public URL

This replaces the current Discord / AirDrop distribution model.
